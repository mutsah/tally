Architecture

One repo, two apps, one database

A NestJS API and a Next.js front end live in an npm-workspace monorepo and share a Postgres database through Prisma. Money is a decimal end to end and is serialised as a string over the wire — it never touches a JavaScript number.

Locked conventions

Money

Decimal @db.Decimal(18,2), serialised as a string. USD only, no currency column, no floats anywhere.

Keys

UUID primary keys throughout. Every domain row carries a userId from the first migration.

Tenancy

Per-user isolation in app code now via a TenantScopedService base. RLS written in v1, activated in v4.

Data model

Six models, every row owner-tagged

The domain is deliberately small. A single Transaction table also carries transfers; manually-valued accounts get their balance from AccountValuation snapshots rather than transaction flow.

Transfer invariant: a transfer is one Transaction with kind = TRANSFER, both accountId (source) and toAccountId (destination) set, and categoryId = null. It is excluded from income and expense at query level, never post-filtered.

schema.prisma

enum AccountKind { CASH BANK SAVINGS INVESTMENT MICROLOANS }
enum CategoryKind { INCOME EXPENSE }
enum TransactionKind { INCOME EXPENSE TRANSFER }

model Account {
  id         String      @id @default(uuid())
  userId     String
  name       String
  kind       AccountKind
  isValued   Boolean     @default(false)   // INVESTMENT / MICROLOANS
  archivedAt DateTime?
  user              User        @relation(fields: [userId], references: [id], onDelete: Cascade)
  transactions      Transaction[] @relation("AccountTx")
  incomingTransfers Transaction[] @relation("ToAccountTx")
  valuations        AccountValuation[]
  @@index([userId])
  @@index([userId, kind])
}

model Category {
  id       String       @id @default(uuid())
  userId   String
  name     String
  kind     CategoryKind
  parentId String?     // one level of nesting only
  parent   Category?   @relation("Tree", fields: [parentId], references: [id])
  children Category[]  @relation("Tree")
  @@index([userId, parentId])
}

model Transaction {
  id          String          @id @default(uuid())
  userId      String
  kind        TransactionKind
  amount      Decimal         @db.Decimal(18, 2)
  occurredAt  DateTime
  accountId   String
  toAccountId String?         // TRANSFER only
  categoryId  String?         // null for TRANSFER
  account   Account  @relation("AccountTx",   fields: [accountId],   references: [id])
  toAccount Account? @relation("ToAccountTx", fields: [toAccountId], references: [id])
  category  Category? @relation(fields: [categoryId], references: [id])
  @@index([userId, occurredAt])
  @@index([userId, accountId])
  @@index([userId, categoryId])
}

model AccountValuation {
  id        String   @id @default(uuid())
  userId    String
  accountId String
  value     Decimal  @db.Decimal(18, 2)
  asOf      DateTime
  @@unique([accountId, asOf])   // upsert-by-date monthly snapshot
  @@index([accountId, asOf])
}

App Router, server-first, BFF auth

Route groups split the unauthenticated shell from the app. A BFF layer proxies the API so the browser never holds a raw JWT. Server data flows through TanStack Query with a deliberate invalidation map.

Quick-add — the signature flow

  • Hero control is reachable from every screen (pinned in the sidebar)
  • Choose Income, Expense, or Transfer — the form adapts
  • Transfer swaps the category field for a destination account
  • Amount is a masked decimal input; kept as a string, never parsed to float
  • On success, invalidate and the dashboard reflects it immediately

TanStack Query — cache invalidation map

MutationInvalidates
create / edit / delete transaction['transactions'] · ['dashboard'] · ['accounts']
record valuation['accounts'] · ['dashboard'] · ['valuations', accountId]
create / edit account['accounts'] · ['dashboard']
mutate category['categories'] · ['dashboard']
A transaction touches balances, so it always invalidates ['accounts'] and ['dashboard'] alongside ['transactions'] — the reason the map is maintained explicitly rather than guessed per call.
The rules money obeys

Balance, transfers, snapshots, export

Derived and valued accounts are separate code paths. Balances come from grouped aggregates, never an N+1 loop. Transfers net to zero across your own accounts.

Balance — flow-derived account

balance =
    Σ income   where accountId = A
  − Σ expense  where accountId = A
  − Σ transfer where accountId   = A  // out
  + Σ transfer where toAccountId = A  // in

Balance — manually-valued account

balance = value of the AccountValuation
          with the greatest asOf
          // distinct on account, order asOf desc
          // transaction flow is ignored

Net worth

Sum of every derived account's computed balance plus the latest snapshot of every valued account. Transfers between owned accounts cancel, so they are excluded from income and expense — net worth is never double-counted.

Dashboard — spending by category

SELECT categoryId, SUM(amount)
FROM "Transaction"
WHERE userId = :me
  AND kind = 'EXPENSE'          -- transfers excluded at query level
  AND occurredAt BETWEEN :from AND :to
GROUP BY categoryId;
-- children then rolled up under their parent in the app layer

Microloan monthly sync

  • Read deployed principal from the external loan system
  • Upsert an AccountValuation for the month (unique on accountId, asOf)
  • Interest received is booked separately as INCOME
  • Principal is never double-counted against interest

CSV export format

  • Custom RFC-4180 writer, not a library dump
  • Columns: date, description, account, toAccount, type, category, amount
  • Money via Decimal.toFixed(2) — exact strings
  • Quotes, commas, newlines properly escaped
